# sdd-prd-check（Raven PRD 预检与需求质量审计）

> 即需求门禁与 PRD 质量检查；与 **design-layer**（UCDP 用例设计）不同，本 Skill 仅做需求侧质量规范，产出 EARS 业务模型与《PRD 质量检查建议》，不产出 UCDP。与 requirement-audit (LATA) 同源，可作为单步预检或串联在流程中。

<context>

## 流程目标与定位

- **核心逻辑**：将非结构化的 PRD 转化为结构化的 **EARS 业务模型**，在不涉及技术实现的前提下识别逻辑漏洞。
- **目标**：产出标准化的《PRD 质量检查建议》，作为后续 AI 生成测试用例的**唯一可信输入**。

## 定位与输入输出

- **阶段**：需求门禁与质量建模 (Design Layer)；本阶段**严禁参考技术域文档**。
- **输入**：业务域文档，或单独输入的 PRD、交互稿、视觉稿（形式：story | prd 地址；纯服务端可无交互/视觉）。**可选**：用户可提供《业务词汇表》；执行技术隔离 (Anti-Tech Leak) 时，若词项命中词汇表则予以保留，不作为缺陷处理（避免业务术语如 `trace_id`、`数据交换平台` 等被误判）。
- **输出**：
  - **必选**：《PRD 质量检查建议》（含评估概览、质量缺陷汇总表、用户故事清单、门禁确认）。
  - **可选**：当非「仅评估」模式时，产出《需求评估过程》（五维解构中间产物），供下游用例设计使用。
- **可独立使用**：是；也可被流程（如 tech-spec-to-testplan Step -1.2）内部调用。

</context>

<rules>

1. **五维深度解构 (5-D Deep Extraction)**：按 **[页面 → 组件 → 属性]** 层级流式扫描，**同步**执行：
   - **流 (Flows)**：主成功路径、分支路径、异常中断/回退路径。
   - **边界与规则 (Boundaries & Rules)**：业务数值区间、身份/权限；**强制 EARS 化**（When... the... shall...）。
   - **状态机 (States)**：业务对象生命周期（如：待审核 → 已发布 → 已下架）。
   - **等价类 (Classes)**：有效/无效输入集合及非法输入处理预期；关注特殊字符、空值、超长字段。
   - **组件颗粒度校验 (Component Audit)**：按 **组件-异常关联映射** 与通用设计 checklist 对齐；识别到具体组件类型时，必须主动检查该类型对应的异常/边界维度。  
     **输入项**：必填性、长度限制、格式正则、报错提示。  
     **展示项**：排序逻辑、分页/加载、**空状态**、默认占位符。  
     **操作项**：点击频率限制、二次确认、前置依赖条件。  
     **示例（搜索框）**：须检查 ① 实时搜索频率限制 ② 特殊字符过滤 ③ 联想词/遮罩层交互；未定义则记入缺陷表。  
   凡 PRD 未定义、表述模糊或逻辑自相矛盾的属性，统一标记为 `[UNKNOWN]`。
2. **EARS 逻辑闭环（逻辑覆盖率）**：解构边界与规则时，必须同时识别**正向条件**与**反向补集**。若定义了 `When [Condition]`，须自检是否存在 `When NOT [Condition]` 或「发生错误时」的处理逻辑；若 PRD 未定义补集行为，**强制标记为 P1 级逻辑空洞**，记入质量缺陷汇总表。
3. **启发式识别**：自动捕捉 PRD 中的 `x`、`?`、`待定` 等占位符号，标记为 `[UNKNOWN]`，并在建议中注明「建议确认是否为后端/运营配置项」。
4. **用户故事清单**：按模块拆分原子化 User Stories，**不包含任何接口/实现细节**。
5. **User Story 可验证性与可追踪性**：每一个 User Story 必须是**可验证的 (Testable)**，且必须**关联至少一条 EARS 规则**；产出用户故事清单时须显式标注或引用对应 EARS，确保需求到测试的 100% 追踪。无法关联到任一 EARS 的 User Story 须标记为缺陷（建议 P2）并建议补充规则或拆解。
6. **技术隔离 (Anti-Tech Leak)**：产出物中严禁出现 `API`、`SQL`、`数据库`、`前端库名` 等术语；若出现须自动订正并生成《需求质量规范 Review — 订正记录》附于文末。**业务术语白名单**：若用户提供《业务词汇表》，命中词汇表的词项予以保留，不作为缺陷处理。
7. **噪音过滤**：默认忽略全局通用技术策略（如：404 页面、统一 Loading、全局网络错误），**仅关注业务特有的异常逻辑**，避免将通用策略误记为业务缺陷。

</rules>

## 文档产出规范：《PRD 质量检查建议》

### 3.1 评估概览

- **需求范围**：覆盖的功能模块/页面索引。
- **核心结论**：2～4 句质量评价。**若无 P0 缺陷，必须写明**：「当前无 P0 阻塞问题，需求完备性良好」。

### 3.2 质量缺陷汇总表 (Merged Audit Table)

**严禁单独设立未知项小节**。合并「未知项」与「质量问题」于同一表，按 **P0 → P1 → P2 → P3** 排序：

| 来源模块 | 缺陷类型 | 详细描述 | 等级 | 建议/改进意见 |
|----------|----------|----------|------|----------------|
| 购物车   | 边界缺失 | 未定义勾选商品刚好达到起送价时的结算逻辑 | P1 | 明确 `>=` 判定 |
| 登录页   | 未知项   | 手机号输入框 `?` 标识 | P2 | 确认是否为后台开关配置 |

可选列：`是否有效`、`是否已改进`（默认「待确认」「否」）。  

**缺陷类型**（表列取值）：**未知项 (Unknown)**：属性缺失、默认值未定义、UI 交互不明。**逻辑矛盾 (Conflict)**：规则冲突、状态机不可逆（未定义回退）。**逻辑空洞 (Logic Gap)**：已定义 When [Condition] 但未定义 When NOT [Condition] 或错误时的处理，记 P1。**边界缺失 (Boundary Issue)**：临界值未闭合、极值/空态/错误态缺失。**合规性缺陷 (Compliance)**：未遵循 EARS、或混入技术实现描述。

### 3.3 状态转移矩阵 (State Transition Matrix)（若存在状态机）

针对识别出的状态机，必须产出 **状态转移矩阵**，便于后续全路径覆盖与非法跳转测试：

- **态分类**：明确 **初始态**、**中间态**、**终态**。
- **合法转移**：列出 PRD 中已定义的允许跳转（如：待审核 → 已发布）。
- **显式禁止转移**：明确指出哪些跳转在业务上**禁止**且 PRD 应约束（例如：已取消订单不可直接跳转至已发货）；若 PRD 未写而由审计推导得出，记入缺陷表并建议补充。

### 3.4 用户故事清单

按模块拆分的原子化 User Stories，不包含任何接口/实现细节。每条 User Story 须**可验证**且**关联至少一条 EARS 规则**（在清单中标注或引用对应 EARS），以支撑需求到测试的 100% 追踪。

### 3.5 门禁确认 (Gatekeeping)

明确列出 **Blocker** 项（P0 级缺陷）；需用户确认「已采纳」或「已修订」后方可推进至下一环节。

### 3.6 缺陷等级定义 (Severity)

| 等级 | 含义 | 典型对应 |
|------|------|----------|
| **P0 (Blocker)** | 业务目标缺失、主流程断点、关键逻辑冲突 | 无法实现主流程、核心功能整段待补充 |
| **P1 (Critical)** | 条件描述含糊、关键异常态缺失、EARS 逻辑不闭合、**逻辑空洞**（缺补集/错误态定义） | 严重一致性冲突、异常态/弱网处理未定义；When A 已定义但 NOT A 未定义 → 标 P1 逻辑空洞 |
| **P2 (Major)** | 边界定义不全、已知未确定项（如 TBD）、配置类缺省 | 需求稿已标「暂未确定」；配置类未写清 |
| **P3 (Minor)** | 组件属性缺失（如缺长度限制）、文案/动效等体验细节 | 组件属性缺省、体验类细节未写清 |

**级别订正**：行为未定义、边界不清晰 → **P1**。仅当主流程整块缺失或断点 → **P0**。需求稿已明确「暂未确定」→ **P2、不着急**。

## 4. 自动化约束与 Review

- **技术隔离 (Anti-Tech Leak)**：严禁出现 `API`、`SQL`、`数据库`、`前端库名` 等术语。若用户提供《业务词汇表》，命中词项予以保留（白名单机制）。
- **噪音过滤**：默认忽略全局通用技术策略（404 页面、统一 Loading、全局网络错误），仅关注**业务特有**的异常逻辑。
- **Review 记录**：若产出物包含技术术语，须自动订正并生成《需求质量规范 Review — 订正记录》附于文末。

## 5. 质量判定标准 (Quality Gate)

| 指标 | 优 (Go) | 劣 (No-Go) |
|------|---------|------------|
| **业务目标** | 目标单一、可量化 | 描述笼统（如「提升用户体验」） |
| **条件描述** | 严格遵循 EARS (When... the... shall...) | 散文式描述、含糊词汇（如「可能」「左右」） |
| **边界完备** | 定义全闭环数值区间与极值/空态/错误态 | 仅常规路径，忽略极值 |
| **技术解耦** | 100% 纯业务逻辑 | 混入「调用后端接口」等实现描述 |

<workflow>

**执行流程**：

1. **五维深度解构与质量检查（一体执行）**  
   仅基于需求稿、视觉稿、交互稿（不引用技术方案）。按 **[页面 → 组件 → 属性]** 层级流式扫描，同步执行流、边界与规则（强制 EARS 化）、状态机、等价类（含特殊字符/空值/超长）、**组件颗粒度校验**（按组件-异常关联映射/checklist，如搜索框→频率限制/特殊字符/联想词遮罩）；**启发式识别** `x`、`?`、`待定` 并标 `[UNKNOWN]`。按 Quality Gate 做质量判定；**噪音过滤**：忽略 404/统一 Loading/全局网络错误，仅关注业务特有异常。所有发现归入**唯一质量缺陷汇总表**，标注缺陷类型与 P0–P3；不单独设「未知项」小节。

   **1.1 逻辑自洽性 Review (Logic-Consistency)**  
   - **补集思维**：针对所有 EARS 语句，若定义了「When [Condition] 则执行 B」，必须自检「When NOT [Condition]」或「发生错误时」是否定义了 C；未定义则记 **P1 级逻辑空洞**。  
   - **状态路径**：识别所有业务对象的状态路径，标记 PRD 中**未定义的非法跳转**（如：已取消 → 已发货），并纳入状态转移矩阵的「显式禁止」或缺陷表。

   **1.2 通用设计指南填补 (Universal Design Guide)**  
   若 PRD 提及 UI 交互但**未详述组件属性**，AI 必须基于 **[通用设计指南]** 自动「填补」潜在风险点并记入检查项；未在 PRD 中定义则标为待确认或缺陷。**示例**：只要有「提交」按钮，必须自动检查「重复点击」与「网络超时」的业务预期（防重复提交、超时提示/重试策略）；若有输入框须检查长度/格式/空值等。将填补项与 PRD 显式对比，缺失即入缺陷表。

2. **结构化梳理**：基于解构结果整理用户故事清单与需求域模块拆分（不填涉及接口）。

3. **产出**：先产出《PRD 质量检查建议》（评估概览 → 质量缺陷汇总表 → **状态转移矩阵**（若存在状态机）→ 用户故事清单 → 门禁确认）。若被流程调用且非「仅需求评估」，再产出《需求评估过程》。

4. **需求质量规范 Review**：检查产出是否含技术术语；若有则订正并生成《需求质量规范 Review — 订正记录》附于文末。

5. **门禁确认 (Gate)**：独立调用时仅交付文档；**流程调用时**必须通过 `ask_user_question` 汇报 P0/P1 风险，等待用户选择 `[已采纳/已修订]` 或 `[需修订]` 后推进或中止。

6. **存储与持久化 (Persistence)**：路径 `/raven/aitest/{需求名}/{日期时间}/PRD_Audit_Report.md`（日期时间可用 `YYYYMMDD_HHMM` 格式，如 `20250312_1430`）。

</workflow>
